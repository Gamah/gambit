# M15 — lichess token key rotation (envelope encryption)

**Status: shipped on `m15-key-rotation`.** Go half compiles and `go test ./... -race` passes on
the dev host (fetch Go 1.22 into scratch). The engine half is untouched — this is a server-only
milestone. Not deployed (no Docker here).

This is the durable milestone doc: the design, the decision that shaped it, and the operator
runbook. The one-paragraph version lives in `CLAUDE.md`'s custody section; PLAN row 95 (rotation)
and row 50 (`code_verifier`) are deleted because they shipped here.

---

## The problem it fixes

Before M15 every stored lichess token was sealed **directly** under one static env key,
`LICHESS_TOKEN_KEY`, and there was **no way to change that key** without turning every token into
an undecryptable row. A leaked key — or any reason to re-key — meant forcing every linked player
to re-link. That was the "no rotation path" open spike (PLAN row 95).

## The decision that shaped it (read this before "hardening" anything)

The instinct is to frame this as confidentiality: "encrypt at rest so a DB dump is survivable."
**On this deployment that is theater, and the design says so out loud.** gamchess and Postgres run
on the same box under the same user, and `LICHESS_TOKEN_KEY` lives in `.env` right there — anyone
who can dump the DB can `cat .env`. So the envelope adds **no secrecy** over the old single key
against the realistic attack (full-box / RCE compromise). The genuine incident lever remains what
it always was: the **audit sweep** + telling lichess our IP is compromised.

What M15 actually buys is **operational, not cryptographic**: the ability to **rotate the data key
on a timer** and **re-key the KEK without orphaning links**. That is the real gap under PLAN row
95, and it is worth having independent of the threat-model debate — a fragile "never touch it or
everyone re-links" secret becomes one the service manages itself. Rotation's confidentiality value
for tokens lichess keeps valid ~a year is marginal; don't re-derive a stronger claim from the
table's mere existence.

> If the token ever moves client-side (the two upstream streaming fixes, PLAN row 15), the whole
> custody problem — and this envelope with it — can be deleted. Nothing here forecloses that.

## The envelope

One **KEK**, many **DEKs**:

- **KEK** (key-encryption key) = `LICHESS_TOKEN_KEY`. Stays a static env secret and the one thing
  to back up. It no longer seals a token directly — it seals the data keys.
- **DEK** (data-encryption key) = 32 random bytes generated by gamchess, stored **only** as
  ciphertext sealed under the KEK (`lichess_key_versions`). The plaintext DEK exists only in
  process memory. Tokens are sealed under the **current** DEK; every `lichess_links` row carries
  the `key_version` that sealed it.

Chain: **KEK (env) → unwraps → DEK (DB, sealed) → unwraps → token (DB, sealed).**

**`key_version = 0` is the legacy sentinel.** Every pre-M15 row gets 0 for free from the migration
default; `OpenToken(.., 0)` opens it with the KEK directly. So the deploy is zero-downtime: old
rows keep working, new links seal under a real DEK (version ≥ 1), and the background sweep drains
the 0s onto the current DEK over time. No data migration runs at deploy.

## What moves, and where

| File | Role |
|---|---|
| `migrations/00003_lichess_key_versions.sql` | the `lichess_key_versions` table + `lichess_links.key_version` (default 0) |
| `internal/keyring/keyring.go` | the `KeyRing`: load/bootstrap, `SealToken`/`OpenToken` (versioned), `Rotate`, `ReEncrypt`, KEK re-wrap, the background `Run` daemon |
| `internal/keyring/ephemeral.go` | `NewEphemeral` + the in-memory `memStore` (DB-less tests) |
| `internal/store/keys.go` | all SQL for `lichess_key_versions` |
| `internal/store/lichess.go` | `LichessLink.KeyVersion`, `LinksNeedingReEncrypt` (cursor-paged), `ReEncryptLinkToken` (optimistic guard) |
| `internal/lichess/crypto.go` | `NewCipherFromBytes` (the AEAD primitive, now used at both KEK and DEK levels) |
| `internal/api/{router,relay,lichess}.go` | `*lichess.Cipher` → `*keyring.KeyRing`; seal/open call sites carry a version |
| `cmd/server/main.go` | reads `LICHESS_TOKEN_KEY_OLD`, `LICHESS_KEY_ROTATION_DAYS` |

`internal/lichess` stays the pure API client (no DB); `internal/store` stays pure SQL (no crypto);
the crypto orchestration lives in the new `internal/keyring`, which sits between them. The keyring
talks to the DB through a small `keyStore` interface so the crypto and rotation logic unit-test
with an in-memory fake — there is no Postgres on the dev host.

## Config knobs (all optional, sane defaults)

| Env | Default | Meaning |
|---|---|---|
| `LICHESS_TOKEN_KEY` | — | the KEK. Blank ⇒ lichess off. Back it up. |
| `LICHESS_TOKEN_KEY_OLD` | blank | previous KEK, set only during a KEK rotation (below) |
| `LICHESS_KEY_ROTATION_DAYS` | 30 | data-key cadence; `0` disables the timer (versioning still works) |

`make up` / `make testinst` still mint `LICHESS_TOKEN_KEY` if absent and **never overwrite** it.

## How rotation runs

`KeyRing.Run` (started from `NewRouter`, process-lived context) does one `maintain` pass at boot —
so legacy `key_version = 0` rows migrate promptly — then one per day:

1. **Rotate** if the current DEK is older than `LICHESS_KEY_ROTATION_DAYS`: generate a new DEK,
   seal under the KEK, `INSERT`, make it current.
2. **Re-encrypt** every row not on the current DEK, in `steam_id` cursor pages. The store update
   is guarded (`WHERE key_version = <the version we read>`), so a player who re-links mid-sweep
   isn't clobbered — their fresh token is already current and the stale re-seal no-ops.
3. **Retire** DEKs that are neither current nor referenced by any row — informational, and it
   stops a drained key being reconsidered as current. Retired rows are **not deleted**: the key
   stays loaded so any stray reference can always be opened.

Old DEKs are kept loaded in memory forever (belt-and-suspenders: no row can ever become
unopenable). That means rotation does not shrink the live-key surface much — deliberate, and
consistent with "the value is re-key-ability, not secrecy."

## Runbook: rotating the KEK (the incident lever)

Because of the envelope, re-keying the KEK rewrites the **handful of DEK rows**, not every token.
To rotate `LICHESS_TOKEN_KEY` (suspected leak, routine hygiene):

1. Generate a new 32-byte key (`head -c 32 /dev/urandom | base64`).
2. In `.env`: move the **current** value to `LICHESS_TOKEN_KEY_OLD`, put the **new** value in
   `LICHESS_TOKEN_KEY`.
3. Deploy. On boot the keyring unwraps each DEK under the new KEK; the ones it can't, it unwraps
   under the old KEK and **re-seals under the new**, persisting the row. Logs: `keyring:
   re-wrapped a data key under the new KEK`.
4. Once the logs confirm the re-wrap, **clear `LICHESS_TOKEN_KEY_OLD`** and deploy again. The old
   KEK is now dead.

No player re-links. If the new KEK can't open a DEK and no old KEK is set, the server **fails at
boot** rather than run half-broken — that's intentional.

## Also in this branch

  floor, right at lichess's undocumented `CodeVerifierTooShort` threshold; 64 bytes → 86 chars is
  margin at zero cost. Removes the "first suspect if linking fails at the exchange" footgun.

## Deliberately NOT done

- **Automatic audit sweep on a timer** was offered and skipped. Auto-hitting `POST
  /api/token/test` for the whole store on a schedule spends the **shared per-IP lichess budget**
  the whole playerbase draws on, with no incident to justify it — exactly the "don't spend the
  shared budget without cause" discipline in `CLAUDE.md`'s etiquette section. The audit stays an
  **operator-triggered** endpoint (`POST /api/v1/lichess/audit`), which is the right shape: you
  run it when you suspect something, not every hour forever.
- **Deleting** retired DEK rows (true crypto-shredding). Kept loaded instead, so no row can become
  unopenable. The threat model doesn't ask for shredding; safety wins.

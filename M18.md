## Context

TV (M9) relays the lichess TV ndjson feed to clients through gamchess, because the
s&box client can't read a raw stream (`Http.RequestAsync` buffers; `RequestStreamAsync`
is broken upstream) â€” so gamchess holds the one lichess stream and clients fetch from it.
That client-facing hop was built as a **version-gated long poll** over a single latest-state
slot, with a stack of machinery layered on top to make a 5s-held poll *feel* like a live
stream: a `since`/`version` cursor, a `hold_ms` field, sawtooth/version-snap guards, and a
full clock latency-compensation apparatus (`clock_age_ms` + `hold_ms` + measured round-trip
+ `LagOf`/`_bankLag` + the one-directional "never read high" house rule). That stack is the
source of the "weird states" and the disproportionate rules/comments around TV.

**M18 replaces the client-facing hop with a WebSocket push.** gamchess still reads the
lichess ndjson (unchanged) and stays the sole token/stream holder â€” this is *not* clients
reading lichess directly (gamchess remains the authority until s&box clients can hold their
own `board:play` tokens). gamchess pushes **one bespoke, self-contained JSON snapshot** to
each connected client whenever a channel's state actually changes. Every message is a full
snapshot ("one packet, all the data") â€” no deltas, no cursor, latest-wins. Clients watching
in-world still get a temporary **fanfare** on game-end, then snap back to whatever is live.

The wire protocol is embraced and documented as a real thing, not apologized for.

### Locked decisions (from the owner)

1. **Transport = WebSocket push**, one bespoke full-snapshot message per state change.
2. **Channel addressing = one socket per channel** (`wss://â€¦/api/v1/tv/{channel}`). Switching
   channels reconnects; the gap is ~one round trip, filled by the stored latest frame pushed
   on connect. Chosen over a multiplexed socket because a client only ever watches one channel
   at a time (so multiplexing is never exercised), and per-channel keeps the wire push-only,
   the `ValidChannel` check at the handshake, and the ref-count leak-proof.
3. **Clock favors reading LOW** ("client early" = player truly has *more* time than shown â€”
   the safe direction, matching the existing house rule). Kept simple:
   - Keep the client's local countdown between moves (lichess only sends a clock on a move;
     a frozen clock reads as broken). Countdown applies **only to the ticking seat**.
   - **Floor the displayed second** (already the case via `TimeControl.Format` truncation) â€”
     the whole-second floor absorbs any sub-second transport latency, so the steady-state
     push reads low for free.
   - Keep **one** field, `age_ms`: gamchess-side staleness, ~0 on live pushes, nonzero only
     when gamchess replays its stored frame to a client that connects mid-think. The client
     subtracts it so a mid-think join doesn't read HIGH (the one case the floor can't cover).
   - Plus a small fixed **`ClockLeadSeconds`** (~0.25s, tunable) undershoot for belt-and-braces.
   - **Delete** `hold_ms`, the round-trip measurement, `LagOf`, `_bankLag`, `MaxLagSeconds`.
4. **Fanfare stays, behavior preserved, code simplified.** Client decides a game ended (its
   featured `game_id` changed); gamchess supplies the reason. Swap pushed immediately, reason
   pushed a beat later after the background `GET /game/export/{id}` returns.
5. **Invariants that do NOT move:** one upstream stream per channel; the `ValidChannel`
   allowlist as a security boundary; the client-facing hop stays session/FP-gated; the lichess
   upstream stays anonymous (no Authorization).

### Traffic sizing (note for cost-awareness â€” put a short version in code/README)

Upstream from lichess is **fixed**: one stream per channel, â‰¤15 channels, independent of
audience. Only gamchessâ†’client egress scales: **~1 KB per move per viewer** (full snapshot),
so a ~40-move blitz game â‰ˆ 40 KB/viewer, ~10â€“15 KB with permessage-deflate. At tens of
concurrent spectators that's single-digit MB/hour; it only matters at thousands of viewers.
TV data is public (anonymous feed, no secrets), so **enabling WS compression is safe here**.

### `server/internal/api/tv.go`

**Reshape `TvState`** (the bespoke wire message â€” one struct, full snapshot):
- Keep: `channel`, `label`, `error`, `game_id`, `url`, `fen`, `last_move_uci`,
  `white_name/black_name/white_title/black_title/white_rating/black_rating`,
  `white_clock/black_clock` (seconds), `ticking_seat`, and the fanfare fields
  `last_game_id/last_status/last_winner/last_white_name/last_black_name`.
- **Rename** `clock_age_ms` â†’ `age_ms`.
- **Delete** `version` and `hold_ms`.
- Keep unexported `clockAt time.Time`; `ageAt(now, reqStart)` â†’ **`stamp(now)`** setting only
  `AgeMs = now.Sub(clockAt)` on the response copy.

**`tvChannel` ref-counting** (replace timestamp scheme with a leak-proof connection count):
- Fields: keep `mu`, `state`, `changed` (the close-and-replace wake primitive â€” the real
  internal signal; the `version` cursor was only ever client-facing), `cancel`. Add
  `conns int` and `lastEmpty time.Time`. **Delete** `version`, `lastPoll`, `touch()`,
  `idleSince()`.
- `update()` / `setLastResult()`: keep bodies **minus the `version++`/`state.Version` lines**.
  `setLastResult` keeps its "drop if `LastGameID != gameID`" guard and its **no-close-on-no-op**
  property (a stale ending must not wake writers / re-snap clients).
- `watch(c)`: lazily open upstream + start sweeper on first arrival; `ch.conns++`.
- `leave(c, ch)`: `ch.conns--`; on 0 set `lastEmpty = now`. **Never cancels** â€” teardown stays
  the sweeper's single job (keeps "we never leak a stream" a property of one function).
- `sweep()`: cancel upstreams where `conns == 0 && time.Since(lastEmpty) >= linger`. Add
  `tvLingerTTL` (~10s, tunable) so an Aâ†’Bâ†’A channel switch doesn't flap the upstream. Repurpose
  the existing idle-TTL/sweep-every constants; keep the TTL a struct field for test override.
- `run` (the frame state machine, swap detection, background `gameResult` fetch): **unchanged**
  except removing the `version++` in `update`/`setLastResult`. The background fetch already
  calls `setLastResult`, which now wakes writers via `close(changed)` â†’ every client gets the
  reason follow-up for free.

**New WS handler `tvSocket` (replaces `tvState`):**
1. `requireSteam` **before** upgrade (normal 401 on failure â€” no upgrade, no upstream).
2. `lichess.ValidChannel(r.PathValue("channel"))` â†’ 404 before anything opens.
3. `h.tvUpgrader.Upgrade(...)`. `CheckOrigin` returns true (client is s&box, not a browser;
   auth is the real gate, Origin is not a boundary here) â€” state this in a comment. Enable
   permessage-deflate (public data, safe; cuts egress).
4. `ch := h.tv.watch(c)`; `defer conn.Close()`; **`defer h.tv.leave(c, ch)`** (guaranteed
   decrement on every exit path â€” the leak-proof core).
5. Reader goroutine: only services control frames / notices the socket dying â†’ `cancel()` so
   the writer unblocks and defers fire (gorilla requires a reader to drain control frames).
6. Writer loop = the old long-poll body, looped, minus timeout/hold/version:
   `snapshot()` â†’ `writeTv` â†’ `select { <-changed / <-ctx.Done() }` â†’ re-`snapshot` â†’ repeat.
   `writeTv` calls `stamp(now)` on the copy (so `age_ms` â‰ˆ 0 on push, real on replay) and
   `conn.WriteJSON`. Snapshot-then-select (grab `changed` under lock before writing) keeps it
   missed-wakeup-free; coalescing is automatic (a slow writer re-snapshots the latest, drops
   intermediates).
7. Ping ticker (~30â€“45s) + read deadline / pong handler â€” classical channels can idle for
   minutes; without keepalive an intermediary (Caddy) may cut the socket.

### `server/internal/api/router.go`

## Risks / unknowns (documented, not denied)

1. `Sandbox.WebSocket.OnDisconnected(status, reason)` may not cleanly distinguish 401 vs 404 vs
   1006 at the handshake â€” the reconnect/`_rejected`/fallback logic depends on telling "channel
   refused" from "auth expired" from "blip". If not exposed, fall back to heuristic (repeated
   immediate closes on a non-default channel â†’ rejected) or a one-shot `Channels()` probe.
2. Whether custom headers beyond `Authorization` (e.g. `X-Steam-Id` for the FP fallback) pass
   `Http.IsHeaderAllowed`. Session bearer self-carries the SteamID in its MAC, so the normal path
   is fine; the pre-session FP fallback over WS may be weaker â€” acceptable.
3. `Connect` is one-shot per instance â†’ every reconnect allocates a new `WebSocket`; ensure old
   instances dispose; backoff guards a reconnect storm.
4. `tvLingerTTL` trades post-leave upstream hold vs channel-switch flapping; correctness doesn't
   depend on the value (guaranteed decrement), only cost.
5. lichessâ†’gamchess leg latency is irrecoverable (nothing downstream knows the move-instant T0),
   so a small residual high-bias survives on the push path â€” `ClockLeadSeconds` is the deliberate
   undershoot that keeps the *displayed* clock low. Magnitude unmeasured, direction certain.
6. Verify Caddy's `wss` proxy idle timeout on the deploy host vs the ping interval.
